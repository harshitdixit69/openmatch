## 1. Project Overview & Philosophy

### 1.1 The Incumbent Problem
The digital matrimonial market in regions like India (traditionally dominated by legacy incumbents such as Shaadi.com and Jeevansathi.com) suffers from severe product degradation caused by extractive and coercive monetization. To enforce monthly subscriptions (often costing $30 to $150), these platforms engage in:
* Forced Friction: Deliberately locking direct communications, blurring user profiles and faces, and restricting access to baseline metrics.
* Extortionate Paywalls: Restricting standard interactions behind monthly or yearly subscription triggers.
* The "Vanishing" Phenomenon: Enabling high-volume low-effort "interest spraying," where users (frequently coordinate or parent-managed) mass-send interests and ghost immediately upon acceptance, rendering high match rates useless.

### 1.2 The OpenMatch Solution: AI Escrow Freemium Model
OpenMatch disrupts this architecture with an "AI Escrow-moderated Fair-Value model" designed natively for mobile:

* Browse & Core Match for Free: Users browse fully-revealed profiles, check compatibility snapshots, and enter a secure real-time message stream for free.
* The AI Escrow Filter: Free chat remains fully functional but is actively moderated by low-latency serverless triggers (openai-gpt-4o-mini). It automatically replaces raw contact exchanges (phone numbers, emails, WhatsApp digits, handles) with a friendly upsell anchor.
* The Mutual Unlock Micro-Transaction: Once two matches establish mutual trust via free escrow chat, they pay a nominal single micro-transaction (e.g., ₹99 or $1) via Stripe/Apple Pay to unlock unmoderated direct phone/WhatsApp calling.
* Intent Escrow & Trust Scoring: Instead of rapid, mindless swiping, OpenMatch requires a friction-budget: siphoning out generic spam by forcing personalized AI-guided reason selections, managing active request limits, enforcing a strict 24-hour reply SLA, and generating dynamic Ghost-Risk and Persona-Type indexes to warn matches of unreliable communications.

## 2. Project Objectives

### 2.1 Product & User Experience Rules
1. Total Transparency: Eliminate blurred profile pictures and obfuscated summaries. Replace with Common Ground Insights (diet, lifestyle, career alignments) and Compatibility Snapshots.
2. Cognitive Safety: Minimize communication anxiety. Expose clear reliability tags (e.g., "Typically replies in 15 mins", "84% Response Rate") before a user invests communication steps.
3. Escrow Fairness: Ensure core matchmaking, discovery, and basic structured conversations are forever free. No high-cost monthly gates.

### 2.2 Technical Requirements
1. Low-Latency Escrow Interception: Ensure message processing, semantic inspection, and PII redaction conclude in < 600ms round-trip to maintain real-time chat responsiveness.
2. Deterministic & Vector-Based Discovery: Match users concurrently via traditional structured SQL preferences (height, age, location) and dense vector representations (pgvector cosine similarity of biographic strings).
3. Ironclad Serverless Orchestration: Implement zero-trust client writing. All match status changes, transaction records, callback systems, and reliability recalculations are processed by serverless triggers to prevent client payload manipulation.

### 2.3 Security & Compliance Targets
* Row-Level Security (RLS): Apply comprehensive Postgres RLS profiles. No client may read another user's contact information, active escrow messages, or billing lines unless explicitly authorized.
* Micro-payments Integrity: Implement atomic, transaction-safe updates. Turn unmoderated paths (is_unlocked = true) only when the backend receives signed and parsed Stripe webhook callbacks.

## 3. Methodology & System Architecture

### 3.1 Technology Stack Selection
The project is built on a modern, decoupled, cloud-native architecture:

* Mobile Client: React Native / Expo (iOS) - Fluid native gestures, dynamic updates, and native Apple Pay support.
* Backend & Storage: Supabase (Postgres) - Robust Postgres indexing, native schema migrations, realtime row subscriptions, and built-in Auth.
* Vector Engine: pgvector - Native distance calculation (<=>) over 1536-dimension user embeddings.
* Serverless Computing: Deno / Supabase Edge Functions - Light execution startup, native TLS, secure environment-secret access.
* Primary LLMs: OpenAI GPT-4o-mini / Cisco Enterprise AI - Best-in-class pricing for semantic interception and structured JSON extraction.
* Realtime Intent: Groq (Llama / Mixtral) - Generates suggestions and intent heuristics under low latency constraints.
* Audio Core: Deepgram Nova / Whisper - Rapid voice note transcription with robust Indian accents support.
* Engagements: Retell AI on Twilio - Automated outbound follow-ups and agent calling for validated matches.
* Payments Gateway: Stripe API - Localized UPI, card processing, and Apple Pay bindings.

### 3.3 Serverless Workflows & Core Algorithms

#### 1. Real-Time Escrow Moderation (The PII Block)
Before writing a message frame during locked states, the proxy calls send-escrow-message:
1. Deterministic Filter: A high-speed regex scan catches basic email structures, @handles, and numeric dial strings.
2. Semantic Parser: The message goes to openai-gpt-4o-mini with a specialized JSON schema. It extracts whether a user is attempting to share PII (even with tricks like writing "nine five six nine" or sending formatted image links).
3. Escrow Redaction: If PII is flagged, the message field in the DB is written with a stylized redaction mark <Upgrade to share contact info> and the UI displays a Stripe mutual-unlock trigger.

#### 2. Dynamic Intent Escalation (Anti-Ghosting SLA)
When an outbound interest request is accepted:
1. interest_requests.first_reply_due_at is set to exactly now() + 24 hours.
2. A database-level daemon checks for first_reply_due_at violations. If the timer hits zero without a message from the sender, the request flips to ghosted.
3. The sender's response_reliability_score is recalculated down, and the daily active_request_limit is throttled as a friction-penalty.

### 4.2 Critical Production Bug Fixes

#### Bug 1: Unscored Quality Sentinel Block (Fixed & Live)
* Incident: Active matches trying to queue trust-callbacks via the Queue AI callback check workflow faced a hard 409 Conflict error:
  ```json
  {"error":"Request quality is too low to trigger an automated callback."}
  ```
* Discovery & Diagnostic: The system checked if request.request_quality_score < 60. However:
  1. The default DB column definition falls back to 0.
  2. The auto-accept thread (auto-accepted upon recipient reply) stored 0.
  3. This meant all valid accepted/ghosted matches that had not undergone manual semantic scoring were locked at 0, triggering a false positive failure.
* Resolution: Updated evaluateCallbackEligibility in trigger-intent-callback/index.ts to ignore 0 as an "unscored" sentinel (meaning the request inherits a passing state because contact trust has been demonstrated by status transitions). In parallel, updated the auto-accept loop in submit-interest-request to write a default 60 quality baseline instead of 0. Both functions were redeployed and verified active.

#### Bug 2: Retell Outbound Phone Number Resolution (Fixed & Live)
* Incident: Placing voice nudges via the AI Broker interface returned a 400 Bad Request from the Retell API:
  ```json
  {
     "error": "Item +919569765152 not found from phone-number",
     "brokerCallId": "f8ad0802-e4be-4b9c-b3cd-5761e2f0a0a8"
  }
  ```
* Discovery & Diagnostic: Two systemic factors:
  1. Caller ID Ownership: The RETELL_FROM_NUMBER environment secret was pointed to a personal destination mobile (+91), but Retell rejects outbound calls unless the sender is an active Twilio/Retell-purchased number registered inside the Retell console.
  2. Destination Target Format: The database profile inputs did not enforce strict E.164 formats, leaving destination numbers saved as raw 10-digit strings (like 9679898989), which Retell rejected.
* Resolution: 
  1. Guided the purchase of an authorized Retell number (+19783818781) and re-anchored the environment secrets cleanly in Supabase.
  2. Rewrote normalizeContactNumber in trigger-outbound-broker-call/index.ts to implement local-national coercion (identifying 10-digit formats, removing prefix 0s, and automatically applying the regional +91 E.164 country code). 
  3. Successfully updated the target database rows via the Supabase Management API. Next-hop calls now dial flawlessly.

#### Bug 3: Webhook Unauthorized and Duplicate Outbound Gating (Fixed & Live)
* Incident: Retell webhook callbacks failed to authenticate (returning a 401 Unauthorized), leaving active broker call attempts stuck in `'queued'` or `'dialing'` states indefinitely. Consequently, subsequent call trigger requests returned a 409 Conflict: `"An active broker attempt already exists for this request"`.
* Discovery & Diagnostic: Two issues:
  1. Webhook Authentication Gap: The generic webhook security checks did not recognize Retell's raw body payload signatures, rejecting the connection as unauthorized before processing completion.
  2. Missing Terminal Cleanup: Because webhooks were rejected, calls never progressed to terminal status (`completed`, `failed`), permanently guarding the request and preventing subsequent attempts.
* Resolution:
  1. Implemented Shared token Authentication: Updated `handle-broker-call-webhook` to authenticate incoming requests via a secure query parameter, verifying `?token=${process.env.BROKER_WEBHOOK_SECRET}` against the configured environment secret.
  2. Implemented Webhook URL Generation: Updated `trigger-outbound-broker-call` to construct a project-unique webhook URL appending the secure verification query token, passing it explicitly in the `create-phone-call` request payload to Retell.
  3. Implemented Auto-Expire Guard: Added a 15-minute expiration block in `trigger-outbound-broker-call`. Any active broker call row stuck in a non-terminal status (`queued`, `dialing`, `in_progress`) for more than 15 minutes is automatically rolled over to `status: 'failed'` with `last_error: 'auto_expired_no_terminal_webhook'`, safely resetting the duplicate gate.
  4. Post-Call Chat Thread Closure: Enabled graceful chat closing by adding `postBrokerClosureMessage`. When a user declines a match to the AI Broker, the Edge Function posts a human-like, plain-language decline text into the chat stream on the user's behalf, providing transparent closure to the counterparty.

#### UI Phase: Plain-Language Conversational Polishing (Fixed & Live)
* Incident: Technical terms like "RETELL", "attempts count", and complex structured alerts ("Availability check in motion" card) cluttered the chat, making the AI-engagement tools feel mechanical and unfriendly to common users. Or, several places duplicated the queue notice.
* Resolution:
  1. Simplified copy/labels: Renamed button actions plainly (e.g. "Check if still interested", "Let OpenMatch call me", "Calls allowed").
  2. Removed dead alert modules: Deleted the double-inserted "Availability check in motion" / `RecoverySuggestionCard` and redundant intermediate `FollowupJobStatusBlock` panels.
  3. Consolidated notices: Compressed the top status warning to a concise note: "Callback queued. We'll update this soon."
  4. Moved AI copilot suggestions to a small "✨ Help" header drawer trigger.

## 5. Monetization & Conversion Strategy

Rather than exploiting user anxiety or hiding profiles behind a paywall, OpenMatch uses a high-margin transactional freemium model.

### 5.1 The Micro-Transaction Unit Model
* The Unlock Unit: Set at an friction-free price point (e.g., ₹99 in India, $0.99 in the US).
* The Psychological Hook: By requiring both users to agree to unlock the match before paying (Mutual Request-Acceptance), users are assured of active interest before completing a payment, eliminating paywall regret.
* Stripe & Apple Pay Integration: Provides seamless native payment sheets right inside the conversation thread.

### 5.2 Conversational Premium Modal Gating (Personalized Targeting)
To capture high-value users (VIPs, parents seeking fast-tracked verification), OpenMatch uses a non-intrusive Premium Modal Popup model instead of static advertisement blocks:
* The Gating Criterion: Modals do not disrupt the first launch of the application. They are scheduled to appear only during active "milestones," such as completing onboarding, sending an outbound interest, or replying within the countdown SLA.
* Warm-User Segmentation (premiumPopup.ts): 
  * Features a gating check that filters out cold or low-intent installations.
  * Measures profile metrics and prior highlight event signals.
  * If a user is categorized as a "Warm" variant (e.g., matches tagged as engaged or high_intent), they receive the popup. 
  * "Starter" variants continue to browse with standard features, keeping the initial experience entirely distraction-free.
* Aggressive Cooldown & Retirement Guard: To protect user experience, any dismissal of the modal increments a local persistent count. If a user dismisses the premium popup twice, the modal self-retires (premium_popup_retired = true), ensuring the app remains clean and respectful.

### 5.3.2 Database Schema Changes (Migration: 20260717210900)

New integer columns added to `public.profiles`:
* `unlock_credits_remaining` — Contact detail unlock credits available
* `super_interest_remaining` — Super Interest sends available
* `spotlights_remaining` — Profile spotlight boosts available
* `subscription_tier` — Active plan identifier (pro / pro_max / pro_supreme)
* `subscription_expires_at` — UTC timestamp for plan expiry

### 5.3.3 Stripe Webhook Credit Depositor (Edge Function: stripe-webhook)

On every successful `payment_intent.succeeded` or `checkout.session.completed` webhook from Stripe:
1. Metadata parsing extracts `userId`, `planTier`, and `durationMonths`.
2. The webhook computes exact upfront credit amounts using the tier×duration multiplier matrix.
3. A single atomic `profiles` update sets all three credit columns and rolls the `subscription_expires_at` forward.
4. The function is deployed with `--no-verify-jwt` to allow Stripe's anonymous callbacks to bypass Supabase's auth gateway.

### 5.3.4 Checkout Session Creator (Edge Function: create-subscription-checkout)

A dedicated Edge Function builds a Stripe Checkout Session for subscription purchases:
* Accepts `userId`, `planTier`, and `durationMonths` from the client.
* Embeds them in Stripe metadata so the webhook depositor can read them reliably.
* Returns a `sessionUrl` for web or a `paymentIntentClientSecret` for native Stripe Sheet.

### 5.3.5 consume_unlock_credit RPC (Migration: 20260718000600)

A server-side PL/pgSQL function (`consume_unlock_credit(p_match_id uuid)`) handles the credit bypass:
1. Verifies the calling user is a participant in the specified match.
2. Checks that the user has an active subscription OR at least one `unlock_credits_remaining`.
3. Decrements the credit atomically.
4. Registers the user's consent in `match_unlocks` with timestamps.
5. If both users have consented, auto-transitions the match to `is_unlocked = true` without any Stripe payment required.

**Bug fixed**: The original parameter was named `match_id`, causing a PostgreSQL 42702 ambiguous-column error when the same name appeared in the `match_unlocks` table. Renamed to `p_match_id`.

### 5.3.6 Dynamic Feature Table UI (MainTabsScreen)

The Premium Upgrade screen now renders a live-updating feature table. Selecting a duration card (1m / 3m / 6m / 12m) instantly recalculates all quota rows (Contact Details, Super Interest, Spotlights) by multiplying the base tier values against the selected duration multiplier, giving the user a clear visual of their bulk credit purchase.

## 5.5 Phase 9: Discovery Feed Filter Fix

### Bug: Null Candidate Attributes Incorrectly Filtered (Migration: 20260718003000)

* **Symptom**: Users with partially-filled profiles (null `religion`, `marital_status`, or `diet`) were invisible in other users' match feeds, even when no explicit preference clash existed.
* **Root Cause**: The `match_profiles` SQL function fell through to a text `LIKE` check on the `preferences` column when a candidate's structured field was `null`. Since most users leave `preferences` blank (`''`), the LIKE check always failed, silently excluding them.
* **Fix**: Updated all three filter branches to treat `null` on the candidate side as a wildcard pass-through:
  ```sql
  -- Before (broken): null → checked preferences text → failed for blank profiles
  or (c.religion is null and lower(coalesce(c.preferences, '')) like '%hindu%')

  -- After (correct): null = not filled in yet = show the candidate
  or c.religion is null
  ```
  Same pattern applied to `diet` and `marital_status`.

## 6. Conclusions & Architectural Roadmap

### 6.1 Architectural Strengths
1. Low Operating Costs: By routing conversational parsing through lightweight serverless edge functions and utilizing optimized semantic payloads, OpenMatch operates at a fraction of the cost of legacy matrimonial backends.
2. Self-Balancing Trust Pools: By using dynamic trust limitations and SLA counters, the platform naturally discourages low-effort spam profiles, fostering a high-quality pool of responsive users.
3. Frictionless Onboarding: A streamlined onboarding flow, combined with beautiful matches, compatibility snapshots, and a fair pay-as-you-go system, dramatically improves registration-to-active conversion rates compared to legacy apps.
4. Bulk Credit Economy: The Phase 9 upfront credit model gives committed users a tangible discount incentive while front-loading revenue, improving cash flow predictability without raising per-unlock pricing.
